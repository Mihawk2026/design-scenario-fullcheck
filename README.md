# Design Scenario Full Check

一个用于 AI 辅助系统设计的 SKILL：基于本地全部历史设计文档，检查当前设计遗漏的业务场景，并按微服务拆分修改点。

普通用户只需要按照下面三步操作，全部通过自然语言完成，不需要配置 RAG、运行脚本、创建数据库或整理 JSON。

## 第一步：安装 SKILL

推荐直接对 Codex 说：

```text
从 https://github.com/Mihawk2026/design-scenario-fullcheck 安装 review-design-impact skill。
```

安装完成后，新建或重新打开一个 Codex 任务，然后输入：

```text
使用 $review-design-impact 告诉我你能做什么。
```

如果能识别 `$review-design-impact`，说明安装成功。

### 手动安装备用方式

下载本仓库，把整个 `review-design-impact` 文件夹复制到 Codex skills 目录：

```text
<CODEX_HOME>/skills/review-design-impact/
```

安装后的目录中应直接包含：

```text
review-design-impact/
├─ SKILL.md
├─ agents/
├─ references/
└─ scripts/
```

不要只复制 `SKILL.md`，否则自动初始化、增量更新和结果校验能力会缺失。

## 第二步：基于历史文档初始化

假设全部历史设计位于 `D:\company-designs`，直接说：

```text
使用 $review-design-impact，把 D:\company-designs 中的全部历史设计初始化为设计场景知识库。
```

SKILL会自动完成：

1. 扫描 Markdown、文本、HTML、JSON、YAML、Word 和 PDF 设计文档。
2. 识别同一个设计的不同版本。
3. 从每份文档提取业务对象、状态、操作、场景和业务规则。
4. 提取每个微服务的参与原因和历史修改点。
5. 建立业务关联和历史共同变更关系。
6. 保存原文路径、版本、章节或页码作为证据。
7. 构建本地历史设计知识。
8. 校验初始化结果。

初始化完成后，应返回类似结果：

```text
扫描设计文档：386份
成功分析：381份
分析失败：5份
历史变更案例：524个
业务场景：2,136个
涉及微服务：67个
需要人工处理：5份无法解析的文档
```

如果文档就在当前工作区，可以不写路径：

```text
使用 $review-design-impact，自动发现当前项目中的全部历史设计并完成初始化。
```

只有找不到设计文档，或者存在多个完全无关的文档库时，SKILL才会询问路径。

### 加入新文档后增量更新

把新增文档放进原来的历史设计目录，然后直接说：

```text
使用 $review-design-impact，增量更新历史设计知识库。我刚加入了新的设计文档。
```

也可以指定新增文档或目录：

```text
使用 $review-design-impact，把 D:\company-designs\2026-Q3 中的新设计增量更新到历史知识库。
```

SKILL会自动比较文件哈希：

- 新文件：分析并加入。
- 内容变化的文件：重新分析并替换旧结果。
- 没有变化的文件：直接复用，不重复分析。
- 删除的文件：移除对应的历史案例。
- 移动或重命名的文件：根据内容哈希和版本关系处理。

每次执行设计检查前，SKILL也会自动检查文档变化。因此不手动说“增量更新”也可以；显式更新适合希望先确认知识库状态的场景。

增量更新完成后，应返回：

```text
新增文档：12份
修改文档：3份
复用未变化文档：381份
删除文档：1份
更新历史案例：24个
失败文档：0份
```

### 初始化数据保存在哪里

SKILL默认在当前工作区维护：

```text
.design-impact/
├─ manifest.json
├─ session.json
├─ cases/
├─ history.db
└─ reports/
```

这是自动管理的内部状态，用户不需要编辑。SKILL不会修改原始历史设计文档。

## 第三步：使用 SKILL 检查设计

### 检查已有设计文档

如果设计文件位于 `D:\current-designs\订单冻结方案.docx`，直接说：

```text
使用 $review-design-impact，检查 D:\current-designs\订单冻结方案.docx 是否遗漏业务场景，并按微服务列出修改点。
```

也可以把设计文档附在对话中，然后说：

```text
使用 $review-design-impact 检查这份设计。
```

### 在设计过程中检查

```text
使用 $review-design-impact，设计订单冻结能力。先分析会影响哪些业务场景和微服务，再输出完整设计。
```

### 只分析影响范围

```text
使用 $review-design-impact，分析“退款支持部分撤销”会影响哪些业务场景、微服务、接口、消息和任务。
```

### 只做微服务拆分

```text
使用 $review-design-impact，把这个需求拆分成每个微服务的修改点、依赖关系、发布顺序和测试项。
```

## 检查结果包括什么

### 1. 场景覆盖矩阵

| 场景 | 当前状态 | 风险 | 责任服务 | 历史依据 |
|---|---|---|---|---|
| 冻结与支付并发 | 缺失 | 高 | 订单、支付 | 历史冻结方案 |
| 冻结期间自动关单 | 部分覆盖 | 高 | 订单、调度 | 历史版本补充项 |
| 重复冻结 | 已覆盖 | 中 | 订单 | 幂等规则 |

场景状态固定为：

- `covered`：已完整覆盖。
- `partial`：提到但方案不完整。
- `missing`：设计没有覆盖。
- `conflict`：和历史业务规则或当前实现冲突。
- `unverified`：有可能涉及但证据不足。
- `not-applicable`：不适用，并提供原因。

### 2. 微服务修改矩阵

| 服务 | 参与原因 | 修改点 | 测试 | 发布要求 |
|---|---|---|---|---|
| order-service | 状态真相源 | 状态机、冻结接口、并发控制 | 状态迁移、并发 | 首先发布 |
| payment-service | 支付许可依赖订单状态 | 支付前校验、错误码 | 冻结后支付 | 兼容新旧状态 |
| scheduler-service | 自动关单读取订单状态 | 排除冻结订单 | 超时关单 | 和订单服务一起灰度 |

### 3. 跨服务完整性检查

- 状态真相源。
- 一致性和最终收敛时间。
- 失败、超时、重试和补偿。
- 新旧版本共存。
- 发布顺序。
- 数据迁移。
- 回滚和人工恢复。
- 日志、指标、告警和审计。

### 4. 证据与待确认项

每个高风险问题都应提供历史设计、业务关联或 CodeGraph 证据。只有通用规则支持的内容会标记为 `unverified`，不会包装成确定结论。

## 完整使用示例

### 首次使用

```text
用户：使用 $review-design-impact，把 D:\company-designs 中的历史设计初始化为设计场景知识库。

SKILL：自动扫描、分析、编译并返回初始化统计和失败文件。
```

### 加入新文档

```text
用户：使用 $review-design-impact，增量更新历史设计知识库。

SKILL：只处理新增、修改和删除的文档，并返回更新统计。
```

### 检查设计

```text
用户：使用 $review-design-impact，检查订单冻结设计有没有遗漏，并按微服务拆分修改点。

SKILL：读取当前设计，使用全部历史案例生成场景覆盖矩阵、服务修改矩阵和高风险问题。
```

## CodeGraph说明

CodeGraph是可选能力。没有 CodeGraph 时，SKILL仍能完成业务场景和微服务影响检查；有 CodeGraph 时，会进一步定位具体接口、消息、表、任务和代码节点。

执行顺序固定为：

```text
历史业务影响分析 → 完整场景和服务清单 → CodeGraph代码定位验证
```

不会因为 CodeGraph 找不到调用关系，就删除历史业务关联发现的候选服务。

## 常见问题

### 是否需要RAG或向量数据库？

不需要。历史文档会被全量结构化，当前需求会和全部历史案例进行条件匹配及业务关联分析。

### 每次检查都会重新分析全部文档吗？

不会。文件哈希没有变化时直接复用，只重新处理新增、修改、删除或移动的文档。

### 初始化时有文档失败怎么办？

SKILL会继续分析其他文档，并列出失败文件和原因。修复文件后说“重新增量更新失败文档”即可。

### 是否会修改历史设计？

不会。所有生成内容保存在 `.design-impact/`。

### 能否在设计生成前使用？

可以。让 SKILL 先分析影响场景和服务，再生成设计，比设计完成后补漏更有效。
